# #3424 — Systemic Root-Cause Analysis: "secret immediately shows 'no longer available'"

Status: Investigation + quick-win observability landed. Companion to the systemic
design spec in [#3496](https://github.com/onetimesecret/onetimesecret/issues/3496)
/ [#3514](https://github.com/onetimesecret/onetimesecret/pull/3514)
(`docs/specs/schema-source-of-truth.md`). This document adds the missing piece
those threads call out: **empirical proof of which fields actually fail**, plus a
concrete inventory of still-live leak sites and the lowest-risk fixes.

Issue: [#3424](https://github.com/onetimesecret/onetimesecret/issues/3424).
Prior fixes that shipped in v0.25.11 but did **not** close it: #3268 (proposal),
#3434, #3477, #3494. Related: #3299, #3496.

---

## 0. TL;DR

- The three prior fixes are **correct and airtight for what they cover**: on
  current `develop`, the recipient reveal payload for a *healthy* secret
  validates cleanly. We proved this by running the real v3 schema against a
  wire-faithful payload — see §2.
- #3424 was never one field. It is a **class** of strict-schema mismatches, and
  the reason it kept "not being fixed" is that **nobody ever captured the field
  that actually fails** — `gracefulParse` discarded it in production. That is now
  fixed (§5, quick win #1).
- The still-live instances of the class are **not on the recipient `ShowSecret`
  path** (which is clean) but on the **receipt / dashboard surface** and on
  **legacy at-rest data**:
  - `ShowReceipt#_receipt_attributes` merges **raw, uncast** `expiration`
    (nullable), `expiration_in_seconds` (raw `secret_ttl`), and `secret_state`
    on top of `safe_dump`, bypassing every numeric cast (§4.1).
  - Receipt timestamp fields `previewed`/`revealed`/`burned`/`shared` are emitted
    **raw** by `safe_dump` — the #3434/#3477 casts never touched them (§4.2).
  - The `viewed→previewed` / `received→revealed` **state rename shipped with no
    data migration**, and `state`/`secret_state` are strict enums that exclude
    the legacy values; `Secret#viewable?` also excludes legacy `viewed`, so a
    legacy-*previewed* secret is bricked into a 404 (§4.3).
- A representative `"no change on v0.25.11"` is *also* fully consistent with the
  deployed image/SPA bundle simply not being the patched build (§3, §6).

A field-level failure matrix, generated by running the real schemas, is in §2.

## 1. The symptom and the four facts that constrain it

A recipient opens a link; the frontend calls `GET /api/v3/guest/secret/:id`; the
backend returns **HTTP 200**; the frontend `gracefulParse(responseSchemas.secret,
…)` fails; `record` stays `null`; the user sees "That information is no longer
available." The sender dashboard shows "Previewed", never "Viewed."

Established by reading current `develop`:

1. **The reveal record is `secret.safe_dump`.** `ShowSecret#success_data`
   (`apps/api/v2/logic/secrets/show_secret.rb:95`) returns
   `{ record: secret.safe_dump, details: {…} }`; V3 inherits it unchanged. There
   is no separate V3 serializer that re-casts.
2. **`safe_dump` hard-casts the four originally-blamed fields.**
   `lib/onetime/models/secret/features/safe_dump_fields.rb:56-65`:
   `lifespan`/`secret_ttl` → `.to_i` (Integer), `created`/`updated` → `.to_f`
   (Float). Even `nil.to_f == 0.0`. These four can no longer be a string or null
   on the wire.
3. **The 200 is guarded by `viewable?`.** `raise OT::MissingSecret … unless
   secret.viewable?` (`show_secret.rb:49`), and `viewable? = (key?(:ciphertext)
   || key?(:value)) && (state?(:new) || state?(:previewed))`
   (`secret/features/secret_state_management.rb`). So when `ShowSecret` reaches a
   parse, `state ∈ {new, previewed}` — both in the enum.
4. **The frontend schema is strict by design.** #3477 *reverted* the
   `z.number().nullable()` widening back to strict `z.number()`
   (`src/schemas/contracts/secret.ts`). The chosen strategy is enforce-at-read,
   clean-at-source — not coerce-on-read.

Facts 1–4 force a conclusion: **for a healthy, current secret the recipient
payload validates** (proven in §2). So the persisting recipient report cannot be
this code path on healthy data — it is legacy/at-rest data, a different endpoint,
or a deployment that isn't running the patched build.

## 2. Empirical proof — the field-level failure matrix

The investigation had been running on inference because production discards the
Zod error. We end that here by doing on our side what production hid: run the
**real** v3 response schemas (`secretResponseSchema`, `receiptResponseSchema`,
`receiptListResponseSchema` — the exact objects `secretStore` validates) against
payloads built to mirror what the Ruby backend actually emits (`safe_dump` +
the raw values merged in the logic classes).

Harness: `src/tests/contracts/issue-3424-failing-field-forensics.spec.ts`
(`npx vitest run src/tests/contracts/issue-3424-failing-field-forensics.spec.ts`).

```
┌─ #3424 field-level failure matrix ────────────────────────────────────────
│ ShowSecret  | healthy fresh secret               PASS
│ ShowSecret  | string lifespan/secret_ttl (pre-#3434) FAIL → record.secret_ttl, record.lifespan
│ ListReceipts| healthy                            PASS
│ ListReceipts| legacy state="viewed"              FAIL → records.0.state
│ ListReceipts| poisoned string previewed          FAIL → records.0.previewed
│ ListReceipts| unset timestamp as ""              FAIL → records.0.revealed
│ ShowReceipt | healthy                            PASS
│ ShowReceipt | expiration=null                    FAIL → record.expiration
│ ShowReceipt | raw string expiration_in_seconds   FAIL → record.expiration_in_seconds
│ ShowReceipt | legacy secret_state="viewed"       FAIL → record.secret_state
└───────────────────────────────────────────────────────────────────────────
```

Reading it:

- **Row 1** is the load-bearing proof: the recipient path is clean for a healthy
  record. **Rows 2** reproduces the original #3424 and shows it is now contained
  to the secret path only when `safe_dump` is bypassed (old backend).
- **Every receipt/dashboard row that fails is a field the per-field casts never
  covered** — legacy enum values, uncast timestamps, and the raw merge in
  `_receipt_attributes`. This is the "next unguarded field" the whack-a-mole
  pattern guarantees.
- Because the schema is an all-or-nothing gate, **one bad row nulls the entire
  receipt list** (the harness pins this too): two good receipts plus one
  legacy-state receipt yields an empty/failed dashboard, not two rows.

This matrix is the artifact #3496/#3514 asked for ("no one has captured the
actual failing field") and the legacy-fixture contract gate they note is missing
(every other v3 schema test uses `state: 'new'` on a pristine record).

## 3. Why three fixes shipped and the bug stayed open

| Fix | What it did | Why it didn't close #3424 |
|---|---|---|
| #3268 | proposed `.to_i` in safe_dump | not merged; superseded |
| #3434 | cast `lifespan/secret_ttl/created/updated` in safe_dump | correct, but only those 4 fields, only the secret/receipt `safe_dump` set — not the raw merges or the receipt timestamps |
| #3477 | write-time coercion (`spawn_pair`, config) + revert contracts to strict | correct, but write-time only protects *new* records via `spawn_pair`; can't heal at-rest data or cover paths that bypass it |
| #3494 | drop `&.` on `created/updated` | correct hardening, same coverage |

All four treated #3424 as a finite list of numeric fields. The matrix shows it is
a class spanning enums, timestamps, and a raw-merge code path. Each fix removed
one or two members of the class; the class remained. And with no observability,
each was validated against `state:'new'` fixtures that can't exhibit the
remaining members.

## 4. The still-live leak sites (verified against current `develop`)

### 4.1 `ShowReceipt#_receipt_attributes` bypasses every cast

`apps/api/v2/logic/secrets/show_receipt.rb:250-265` starts from
`receipt.safe_dump` and then `merge!`s raw values:

```ruby
expiration: expiration,                       # = receipt.secret_expiration → nil when secret_ttl is falsey
expiration_in_seconds: expiration_in_seconds, # = receipt.secret_ttl       → RAW (no .to_i)
secret_state: secret_state,                   # = secret.state             → raw, can be nil/legacy
```

The v3 contract (`src/schemas/contracts/receipt.ts:234-235`,
`shapes/v3/receipt.ts:72`) requires `expiration: z.number()` (non-null) and
`expiration_in_seconds: z.number()`. A consumed/expired secret yields
`expiration: null`; a poisoned record yields a string `expiration_in_seconds`.
Both reject. Matrix rows 8–10 confirm.

### 4.2 Receipt timestamp fields are emitted raw

`receipt/features/safe_dump_fields.rb` emits `previewed`/`revealed`/`burned`/
`shared` (and the compat `viewed`/`received`) with **no lambda** — the raw stored
value. The v3 shape maps them through `transforms.fromNumber.toDateNullish` =
`z.number().nullish()`, which accepts a number/null/undefined but **rejects a
numeric string and rejects `""`**. The #3434/#3477 casts only ever touched
`lifespan/secret_ttl/created/updated`. Matrix rows 5–6 confirm.

There is a hydration hinge here worth a backend test: Familia returns `""` for
some unset/loaded fields. If an unset `previewed`/`revealed` serializes as `""`
rather than `null`, those rows fail *universally*, not just for poisoned data
(matrix row 6). The benign-vs-broken dashboard turns on this single behavior.

### 4.3 The state rename has no migration, and `viewable?` excludes legacy `viewed`

The `'viewed'→'previewed'` / `'received'→'revealed'` rename is documented in the
"MIGRATION SCRIPT REQUIREMENTS" comment blocks in both `safe_dump_fields.rb`
files — **but those are comments; no migration exists** under `migrations/`. The
strict enums (`receiptStateSchema`, `secret_state`) exclude the legacy values, so
any pre-rename receipt fails the list/show schema (matrix rows 4, 7, 10).

Worse, `Secret#viewable?` accepts only `new`/`previewed`, **not** legacy
`viewed`. A secret that was merely *previewed* (link opened, never consumed) on a
pre-rename build is therefore bricked into a `404 → "no longer available"` after
upgrade, even though its value is intact. This is a second, independent path to
the recipient symptom — and being a 404, it renders through the *same*
`UnknownSecret` view as a parse failure, which is why the reporter (reasonably)
read it as a parse failure.

## 5. Root causes (verified), and what each fix must address

Building on the seven in #3514, confirmed against code and sharpened with the
matrix:

1. **Type-preserving, not type-enforcing storage (Familia v2).** A value written
   as the wrong Ruby type round-trips forever; no `field :x, type: Integer`.
2. **Open-ended set of string-injecting write paths** (ERB/ENV, YAML floats, raw
   params/console). #3299 closed two; without a single choke point the next is
   inevitable.
3. **A strict, all-or-nothing read contract.** One bad field nulls the whole
   record/list → `UnknownSecret`. Authored to the idealized model, not to what
   `safe_dump` + logic merges actually emit across the real keyspace.
4. **Remediation at the wrong layer (per-field casts).** `safe_dump` casts are
   correct but incomplete by construction; `_receipt_attributes`'s raw merge and
   the raw receipt timestamps are living proof.
5. **A broken diagnostic loop.** The failing field was discarded in production —
   the direct reason three fixes were shipped on inference. **(Fixed in §6.)**
6. **An ambiguous failure surface.** A real 404 (expired/consumed/legacy-bricked)
   and a schema parse failure render identically as `UnknownSecret`.
7. **No data migration for the state rename**, plus `viewable?` not accepting
   legacy `viewed` (§4.3) — turns legacy data into both 404s and enum failures.
8. **Deployment/version skew.** Helm pins an image; the SPA bundle can be
   CDN/browser-cached. "No change on v0.25.11" is consistent with the running
   build not actually being v0.25.11. Confirm before assuming a code defect (§7).

## 6. What landed in this branch (quick wins)

### Quick win #1 — make the failure self-reporting (root cause #5)

`src/utils/schemaValidation.ts`: `gracefulParse` now builds its message **from
`error.issues`** (path + code + Zod's type-level message) and promotes the failing
paths to a **searchable Sentry tag** `schemaField` (registered in
`src/services/diagnostics.service.ts` `TAG_FIELDS`). Only field paths and codes
are logged — never values — so it is safe for secret payloads.

Effect: the next report carries `record.expiration` or `records.3.state` instead
of a generic string. This is the prerequisite that lets us *prove* a fix closes
#3424 rather than infer it, and it costs nothing in risk.

### Quick win #2 — the empirical contract gate (root causes #3, #4)

`src/tests/contracts/issue-3424-failing-field-forensics.spec.ts` runs the real
schemas against legacy/poisoned/raw-merge fixtures (the gap §3 names) and pins
the matrix in §2 so any regression in `safe_dump` shape or contract strictness
fails CI with the field named.

Both are additive and low-risk; neither changes a wire payload or a contract.

## 7. Recommended next steps (in priority order)

**Immediately, for the reporter (operator diagnostics, no deploy):**

- Confirm the running build. `GET /api/v3/guest/secret/:id` against a fresh
  secret and check whether `record.lifespan` is a JSON number vs string, and
  whether the response even contains `is_previewed`. If string/missing, the pod
  is not the patched build (root cause #8) — re-pull/clear cache first.
- Pull raw bytes: `redis-cli HGET secret:<objid>:object lifespan`. `"604800"`
  *with quotes* = a string was written upstream (write-boundary/legacy);
  `604800` = healthy.
- Scan for legacy state and poisoned numerics with
  `scripts/diagnostics/detect_string_typed_numerics.rb` (extend it to also flag
  out-of-enum `state` — see below). The "Previewed but never Viewed" symptom maps
  to legacy `state` on the receipt list path.

**Short term (close the live leaks the matrix found):**

- Fix `_receipt_attributes` (§4.1): `expiration_in_seconds` → `.to_i`; decide the
  `expiration: null` contract (a consumed/expired receipt legitimately has no
  expiration — the contract should be `z.number().nullable()` here, *or* the
  logic should emit a sentinel; this is the one place strict-non-null is wrong).
- Cast/normalize the raw receipt timestamp fields (§4.2), and resolve the `""`
  hydration hinge with a backend tryout.
- Add the missing `viewed→previewed` / `received→revealed` migration under
  `migrations/`, and make `Secret#viewable?` (and the receipt enums) accept
  legacy `viewed`/`received` until the migration has run everywhere (§4.3).

**Systemic (the real cure — adopt #3496/#3514):**

- Contracts as the single model source of truth; generate `storage/` and
  `shapes/` JSON Schemas; validate `to_h` pre-save and `safe_dump` pre-response
  with Familia's `schema_validation`. This makes drift a build/test failure
  instead of a production incident, and replaces every per-field cast (including
  the ones added for #3424) with generated coercion.
- Fail-soft the consumer (Phase 5): a single malformed non-essential field
  should degrade that field, not hide a viewable secret, and `UnknownSecret`
  should distinguish a true 404 from a parse failure (root cause #6).

The MVP that actually closes #3424 = observability (landed) + pre-response
validation + generated coercion incl. `state` normalization + the state
migration. Per-field casting alone will not, by construction.
