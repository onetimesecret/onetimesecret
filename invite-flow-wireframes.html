OTS Invite Flow Wireframes

State 1: New User (no account for invited email)

Invite details and account creation on one screen. Signup + accept collapsed into a single action.

Both paths: identical layout, branding source differs
A Email field is read-only, pre-filled from the invite. The invite is email-bound; there's nothing to edit. Displaying it reinforces "this is your identity here."
B Auth method tabs reflect the org's configured auth methods, not the global OTS defaults. If the org only allows SSO, only SSO shows. If password + magic link, both show.
C Single primary action: "Create Account & Join". No separate accept step. The account is created and the invite accepted atomically.
D Decline is present but de-emphasized. Declining here means the invite is marked declined (visible to admin in audit log) and the user is not redirected to signup.

State 2: Existing User, Not Signed In

Account exists for invited email. User must authenticate to prove ownership, then auto-accept.

Same layout both paths
A The heading changes to "Sign in to join" rather than generic "Sign in to your account." The invite context stays visible throughout auth, so the user never loses sight of why they're here.
B Email is read-only. No "Create account" link. The system knows this account exists (resolved at page load from the invite token). The user can't switch to a signup flow or change the email.
C On successful authentication, auto-accept the invite and redirect to the workspace. No intermediate confirmation screen. The user already saw the invite details; authenticating is the consent act.
D Password reset must preserve the invite token. If the user clicks "Forgot password," the reset flow must return them here, not to a generic login page. This is the fragile state-preservation point from the analysis.

Identity matches. Minimal friction. Show details, single action.

Same layout both paths

B Shows who they're signed in as explicitly, so there's no ambiguity about which identity is accepting.
C Decline is a first-class action here (not just a footer link). At this point the user is making a deliberate choice, not navigating auth machinery.

State 4: Signed In as Wrong Email

Email-bound invite. No "accept with this account" option. The user must authenticate as the invited email.

Same layout both paths
A The mismatch is stated as fact, not error. The framing isn't "wrong account" (which implies the user did something wrong). It's "this invite is for X, and you're Y."
B Single action: "Continue as [invited email]". This signs the user out of the current session and reloads the invite page, which now renders as State 1 (no account) or State 2 (account exists). No "Sign in with different account" framing, because there's no choice involved. The invite determines the email.
C The invited email is shown prominently. The current session email is shown for context. The user can see both and understand the situation without interpreting an error code.
D Decline is still available. The user might have received this invite by mistake, or they might not want to create/use a separate account for this email.

State 5: Already a Member

Edge case. Ideally blocked at invite-send time. If it reaches the user anyway: non-error, soft redirect.

Same layout both paths
A Not an error. The current flow shows a red "Error" banner with a fire icon. This is a successful state (user is already where they need to be). The tone should reflect that.
B If the invite specifies a different role than the user's current membership, that's noted here. Whether it triggers a role change is a separate backend decision, but the user should see the discrepancy rather than have it silently swallowed.
C Single action: go to the workspace. No Accept/Decline buttons (there's nothing to accept).

State 6: Expired or Revoked

Terminal states. Indistinguishable to the recipient (no information leak on revocation). Clear next step.

Canonical site only (org branding may not load if invite is fully expired from DB)
A Same message for expired and revoked. "This invitation is no longer valid." Revoked invites don't reveal that someone actively cancelled them.
B If the inviter's email is available from the (expired) invite record, show it as the contact point. If the invite record is fully purged, fall back to generic "contact your administrator" language.
C No Accept/Decline buttons. No auth form. This is a dead end with a clear escape hatch.
